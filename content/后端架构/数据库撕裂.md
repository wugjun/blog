## 1. 根源：扇区与页的大小错位

数据库以 **页 (Page)** 为单位思考（通常 16KB），硬盘却以 **扇区 (Sector)** 为单位存储（通常 512 字节）。  
写入一个页需要依次写入 **32 个连续扇区**。如果在写入中途断电：

- 前 17 个扇区：新数据 ✅
- 第 18 个扇区：正在写入 → 数据损坏或未定义 💥
- 后 14 个扇区：旧数据 🕒

结果：一个逻辑页在物理上变成了“新 + 坏 + 旧”的混合体 —— **页撕裂 (Torn Page)**。

> 注：单个扇区的写入在硬件上具有原子性（要么全写对，要么全不写），但跨扇区无此保证。

## 2. 如何发现撕裂？页校验 (Page Checksum)

数据库在写入页前，会根据整个页内容计算一个 **校验值 (Checksum)** 并存入页头。  
读取时重新计算并与页头比对：

- 若页因撕裂而新旧混合 → 新校验值 **几乎必然 ≠** 旧校验值
- 数据库报错：`Checksum mismatch`，明确告知数据损坏。

## 3. 传统软件层解法：写两遍

硬件不提供跨扇区的原子性，数据库就在软件层做冗余。

### 3.1 MySQL / InnoDB：双写缓冲区 (Doublewrite Buffer)

1. 脏页不直接写到最终位置。
2. 先**顺序写入**一个专用区域 —— 双写缓冲区。
3. 调用 `fsync()` 强制落盘。
4. 再随机写回真实位置（B+Tree）。
5. 崩溃恢复时：若真实页校验失败，从双写缓冲区拷贝完整副本覆盖。

代价：**写放大 2 倍**。

### 3.2 PostgreSQL：整页写入 (Full-Page Write)

1. Checkpoint 后，某个页首次被修改时。
2. 将完整的 **8KB 页** 写入 WAL (Write-Ahead Logging，预写日志)。
3. 同一周期内的后续修改只记录增量（字节级变化）。
4. 恢复时：WAL 中的整页镜像 + 增量回放。

代价：WAL 体积峰值放大 **约 14 倍**。

## 4. 硬件进步：Linux 6.13 块原子写

Linux 内核 6.13 引入真正的块层原子写支持，让硬件做出承诺：**一个逻辑写操作要么整体落盘，要么完全不落盘**。

- 新增 `RWF_ATOMIC` 标志（`<linux/uio.h>`）。
- 新增 `statx` 字段查询设备原子写能力。
- 块层定义 `REQ_ATOMIC` 语义：**绝不允许拆分该 bio**。
- 统一了 NVMe、SCSI 等 5 种硬件的底层实现。

## 5. 数据库如何使用原子写

应用程序需要主动适配，步骤：

1. **探测能力**：调用 `statx()` 获取设备的原子写限制，回填：
   - `atomic_write_unit_min`：最小原子单元
   - `atomic_write_unit_max`：最大原子单元
   - `atomic_write_max_segments`：最大连续段数
2. **对齐分配**：使用 `fallocate()` 分配磁盘空间时，保证一个原子写不跨越两个连续 **extent**（物理不连续 ⇒ 硬件无法保证原子性）。
3. **发起原子写**：调用 `pwritev2(fd, iov, iovcnt, offset, RWF_ATOMIC)`。
   - 若满足硬件约束，写入成功 ⇒ 数据库可关闭双写缓冲区和整页写入。

## 6. 原子写的四条硬约束

对 `RWF_ATOMIC` 的调用必须同时满足：

| 约束                            | 说明                  | 原因                                                                                                       |
| ------------------------------- | --------------------- | ---------------------------------------------------------------------------------------------------------- |
| **总长度是 2 的幂**             | 如 4KB、8KB、16KB     | 避免跨 NVMe NABSPF (Namespace Atomic Boundary & Size Parameter Fallback) 边界，否则硬件会拆分为两个独立 IO |
| **长度 ∈ [unit_min, unit_max]** | 由设备决定            | 超出硬件能力则无法保证原子性                                                                               |
| **偏移自然对齐**                | `offset % len == 0`   | 确保起始位置与原子单元边界对齐                                                                             |
| **不跨物理不连续区域**          | 依赖 `fallocate` 保证 | 硬件原子写要求逻辑块连续且物理连续                                                                         |

## 7. 内核实现：queue_limits 与 REQ_ATOMIC

内核块层在 `queue_limits` 结构中维护原子写专用参数：

```c
struct queue_limits {
    unsigned int atomic_write_unit_min;
    unsigned int atomic_write_unit_max;
    unsigned int atomic_write_boundary;   // 映射 NVMe NABSPF
    unsigned int atomic_write_max_bytes;
};
```

核心铁律：**带 `REQ_ATOMIC` 标记的 bio 绝不允许拆分**。  
若下层设备需要拆分，则拒绝该 IO（返回错误），而不是悄悄拆成多个独立 IO —— 因为一旦拆分，掉电可能只成功一部分，原子性即破裂。

## 8. 硬件如何真正保证原子写？

以 NVMe SSD 为例（使用 Direct I/O）：

- **AWUPF** (Atomic Write Unit Power Fail)：掉电原子单元大小，SSD 保证该单元内的写入断电时要么全完成、要么全不完成。
- **NABSPF** (Namespace Atomic Boundary & Size Parameter Fallback)：原子边界对齐参数。
- **内部实现**：SSD 的 FTL (Flash Translation Layer，闪存转换层) 采用**整体映射表切换** + **掉电保护电容**。写入时准备新映射表，断电后电容提供足够电量完成当前操作，保证原子边界内的完整性。

## 9. 性能提升：实测数据

PostgreSQL 开启原子写后的效果（对比旧方案）：

| 指标             | 提升                                       |
| ---------------- | ------------------------------------------ |
| TPS (每秒事务数) | **提升 1.7 倍**                            |
| WAL 写入量       | **降低约 14 倍**（因为无需再写整页到 WAL） |
| 写延迟波动       | 大幅减小                                   |

值得一提的是，之前 **一半的 PostgreSQL 用户运行在 Buffered I/O 上**（以规避双写开销），但缓冲 I/O 存在撕裂风险。原子写让 Direct I/O 成为更安全且高效的选择。

## 10. Direct I/O vs. Buffered I/O 与原子写的关系

| 特性           | Direct I/O                       | Buffered I/O (Page Cache)              |
| -------------- | -------------------------------- | -------------------------------------- |
| 路径           | 应用缓冲区 → 直接发给硬件        | 应用缓冲区 → 先到内存页缓存 → 异步刷盘 |
| 同步性         | 全程同步，一次到底               | 异步，`write()` 返回时可能未落盘       |
| 缓冲区稳定性   | 应用需自行保证 IO 期间缓冲区不变 | 内核允许修改，存在撕裂风险             |
| 对齐要求       | 应用必须自己处理对齐             | 内核可以处理非对齐，但原子写无法保证   |
| Linux 内核支持 | 6.11 / 6.13 已文档化支持         | 目前存在争议（见下节）                 |

## 11. 前沿争论：2026 LSFMM 关于缓冲原子写的讨论

在 Linux Storage, Filesystem, Memory-Management 峰会上，内核开发者讨论了是否让 Buffered I/O 也支持原子写。

**提出的方案**：

- **writethrough + RWF_ATOMIC**：`pwritev2` 时立即发起 I/O，不等后台回写。
- **stable pages**：I/O 期间给物理页打短期 pin (固定)，禁止修改。
- 组合标志：`RWF_WRITETHROUGH` 可能需要与 `RWF_ATOMIC` 一起使用。

**争议点**：

- `dsync` 是强保证，`writethrough` 只是弱保证。
- 与其增加 Page Cache 的复杂度，不如推动数据库全面走向 **Direct I/O**。
- 建议用 **io_uring** 承载原子写需求，而不是无限增加 `RWF` 标志。

## 12. 一次原子写的完整 IO 路径（时间线）

以下为 PostgreSQL 在支持原子写的 SSD 上写入一个 8KB 脏页的耗时分解：

| 步骤 | 操作                                            | 耗时          |
| ---- | ----------------------------------------------- | ------------- |
| 0    | Checkpoint 后首刷脏页（旧路径需要写整页到 WAL） | 0 µs (已省去) |
| 1    | `pwritev2` 带 `RWF_ATOMIC` 标志下发             | 2 µs          |
| 2    | VFS 校验四约束（2 的幂、区间、对齐、段数）      | 3 µs          |
| 3    | 块层构造 bio，打 `REQ_ATOMIC` 标记，绝不可拆分  | 8 µs          |
| 4    | NVMe 驱动转为 Write 命令，落在单个 AWUPF 单元内 | 12 µs         |
| 5    | SSD FTL 映射表整体切换：全成功或全不成功        | 15 µs         |
| 6    | 返回成功，全程没有写第二遍                      | 80 µs (总)    |

> 对比旧方案：双写缓冲区需要先顺序写双写区（`fsync`），再随机写回真实位置，总延迟通常在数百微秒到毫秒级。

## 13. 总结：撕裂的根源是粒度的错位

- **数据库以页为单位思考，硬盘以扇区为单位存储** —— 两者粒度不一致，跨扇区的写入没有硬件原子性保证。
- 过去 30 年，数据库在**软件层用两倍写入**（双写缓冲区 / 整页写入）来硬抗硬件的不足。
- **原子写**让硬件做出明确承诺：要么整个逻辑单元一次性落盘，要么完全不落盘。
- 当这一承诺从应用程序 (`RWF_ATOMIC`) 经由 VFS、块层、NVMe 驱动，一路传递到 SSD 的 FTL 时，那一半的冗余写入就被**永久地省掉了**。

> 这不是修修补补，而是上下层对齐粒度后的质变。
