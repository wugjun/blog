🧵 ThreadLocal 实现内幕

全局共享 vs 线程私有 · 开放寻址 · 黄金分割哈希
🌐 全局共享 Map 早期设计
  • 单个 ThreadLocal 持有全局 Map<Thread, T>
  • 所有线程用同一张表，Key 为线程引用
  • 多线程读写 必须加锁 / CAS
  • 线程死亡后，Entry 仍残留在 Map 中 → 可能内存泄漏
⚠️ 全局竞争 + 跨核 Cache 乒乓
🧩 线程私有 Map 当前主流 (JDK8+)
  • 每个线程持有自己的 ThreadLocalMap
  • Key 为 ThreadLocal 实例，Value 为存储值
  • 零同步：各线程只读自己的 Map
  • 线程死亡 → 整个 Map 及 value 一次性回收
  • 访问路径：currentThread().threadLocals，一次字段读取
✅ 无锁、缓存友好、生命周期自然绑定

🔍 开放寻址 + 线性探测 Open Addressing
  • 不使用 红黑树或链表（Entry 数量预期极少）
  • 冲突后顺数组向后找空槽，缓存行友好，优于指针追溯
  • Entry 结构轻量，无额外指针开销

✨ 魔数 0x61c88647 · 黄金比例 Fibonacci 哈希
private static final int HASH_INCREMENT = 0x61c88647; // 2^32 * 0.618 ≈ 1640531527
  • 每个新 ThreadLocal 的 hash 按此递增：nextHashCode += HASH_INCREMENT
  • 容量为 16 时，低 4 位等价于每次 +7（因 7 与 16 互质）
  • 16 次插入恰好不重复地走遍全部槽位，避免冲突

⚡ 零同步 · 缓存友好
  • 线程只读自己的 Map，不存在竞争路径
  • 若用全局 Map<Thread,T>，则必须加入并发控制（锁/CAS）
  • 访问路径仅 currentThread().threadLocals，无哈希查找
  • 避免跨核缓存一致性开销
♻️ 生命周期 · 内存安全
  • 线程死亡 → Thread 对象不可达 → 整个 Map 及所有 value 一次性回收
  • 数据生命周期天然绑定线程，而非绑定可能永生的静态字段
  • 消除手动清理 remove() 的遗忘风险
⚡ 基于 JDK 8+ ThreadLocal 实现  ·  可视化仅供学习参考
